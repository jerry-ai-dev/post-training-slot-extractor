# 阶段一：工程脚手架、推理 Harness 与评估框架

## 1. 阶段目标

阶段一建立可复现、可测试、可扩展的本地后训练工程骨架。交付重点不是模型质量，而是打通从评估样本、Prompt 组装、模型推理、结构化解析、规则评分到报告落盘的完整链路。

阶段完成后，后续工作只需要扩充数据、评分规则、训练流程和部署配置，不需要重新搭建基础工程。

## 2. 工程结构

核心目录职责如下：

| 目录 | 职责 |
|---|---|
| `configs/` | 推理与评估配置 |
| `data/` | 冻结评估集、训练数据和数据合同 |
| `deployment/` | 本地推理服务与部署辅助文件 |
| `experiments/` | 训练和对比实验输出 |
| `models/` | 基座、Adapter、合并权重与 GGUF 文件 |
| `reports/` | 可保留的正式评测报告 |
| `scripts/` | 数据校验、评测、训练和部署入口 |
| `src/slot_extractor/` | Schema、Prompt、推理和评估实现 |
| `tests/` | 单元测试与集成测试 |

项目使用 Python 3.12、`uv`、`pyproject.toml` 和 `uv.lock` 固定依赖环境。大模型权重、运行缓存和临时产物不进入版本库。

## 3. 核心契约

### 3.1 评估样本

评估样本由 `Sample` 表示，包含唯一 ID、场景层级、运行时输入、标准输出、可执行断言、场景标签，以及可选的回复语义要求和多轮链路信息。

加载入口负责验证必填字段、字段类型、自然语言 History 和多轮链路结构，错误数据在进入推理前直接失败。

### 3.2 模型输出

模型输出只有两种动作：

- `tool_call`：请求调用 `find_technicians`，并给出完整参数；
- `final`：返回当前预约状态、流程状态、回复类型和面向用户的回复。

输出使用严格 JSON 合同，不接受 Markdown 代码块、解释性前后缀或额外字段。解析失败作为协议错误记录，不由评测代码猜测或修补。

### 3.3 推理结果

每条推理结果保留原始模型文本、解析后的结构化输出、Prompt 消息快照、首字和总时延、输出 token 数，以及各评分器结果与错误信息。

这些字段使评测可复现，也便于定位问题属于数据、Prompt、模型、推理服务还是评分器。

## 4. Prompt 组装

`PromptBuilder` 统一生成模型消息，避免脚本各自拼接提示词。Prompt 包含：

1. 角色与任务边界；
2. 严格输出 Schema；
3. 状态继承、覆盖和工具结果有效性规则；
4. 当前时间、历史、当前状态和工具上下文；
5. 可用工具定义；
6. 当前用户输入。

Mock 与本地模型共用同一套 Prompt 入口，测试、评测和真实推理不会形成不同合同。

## 5. 推理后端

推理后端通过统一协议接收消息并返回文本与时延数据。评估 Runner 不感知具体模型来源，只依赖后端接口。

### 5.1 Mock 后端

Mock 后端按样本 ID 返回确定性结果，用于验证完整管道、测试输出解析和评分、执行快速回归，以及构造稳定的边界与失败案例。

### 5.2 llama.cpp 后端

本地后端调用 `llama-server` 提供的 OpenAI 兼容接口。配置包含模型标识、服务地址、温度、最大输出长度和超时时间。默认使用确定性采样设置，保证同一模型和配置下的结果尽量稳定。

## 6. 评估管道

评估流程固定为：

```text
加载并校验样本
→ 组装 Prompt
→ 调用后端
→ 解析严格 JSON
→ 执行评分器
→ 聚合分数与时延
→ 写入报告
```

评分器实现统一接口，每个评分器返回分数、是否适用和诊断详情。聚合器区分有效分数与 `N/A`，不会把尚未实现或不适用的维度当成零分。

阶段一配置保留基础协议、字段、工具调用和速度检查，用于证明框架可以同时处理质量与性能数据。

## 7. 测试策略

测试分为三层：

- 单元测试：Schema、JSONL、Prompt、后端、评分器和分数卡；
- Mock 集成测试：完整执行样本到报告的确定性链路；
- 本地模型集成测试：连接真实 `llama-server`，验证请求、响应和计时链路。

本地模型测试使用独立标记，常规 CI 或无模型环境可以跳过，不影响其余测试。

## 8. 阶段交付

阶段一最终交付：

- 可安装、依赖锁定的 Python 项目；
- 严格的样本、输出和结果 Schema；
- 单一 Prompt 组装入口；
- Mock 与 llama.cpp 两种推理后端；
- 插件式规则评分与分数卡；
- 单元、Mock 集成和本地模型集成测试；
- 可供阶段二直接扩展的评估管道。
